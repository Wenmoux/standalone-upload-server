# Docker deployment

This project can run as four containers:

- `postgres`: local PostgreSQL storage.
- `server-pg`: API/admin service on host port `3100`.
- `reader`: `cirno-src` reader service on host port `3200`.
- `bot`: Telegram bot process, with a localhost-only health port `3300`.

## Configure

Create a private `.env` from `.env.docker.example` and replace all `change-this` values:

```powershell
Copy-Item .env.docker.example .env
```

Required values for real deployment:

- `POSTGRES_PASSWORD`
- `PO18_PG_URL`
- `PO18_UPLOAD_ADMIN_PASSWORD`
- `PO18_UPLOAD_SESSION_SECRET`
- `PO18_UPLOAD_API_TOKEN`
- `PO18_BOT_API_TOKEN`
- `TELEGRAM_BOT_TOKEN`

Optional bot export values:

- `PIKPAK_WEBDAV_URL`
- `PIKPAK_WEBDAV_USERNAME`
- `PIKPAK_WEBDAV_PASSWORD`
- `PIKPAK_WEBDAV_ROOT`

## Start

```powershell
docker compose up -d --build
```

To run the published Docker Hub images instead of building locally:

```powershell
docker compose -f docker-compose.hub.yml up -d
```

`docker-compose.hub.yml` uses one application image, `wenmoux/reader:v1.0`, for `server-pg`, `reader`, and `bot`; each service only overrides `command`.

## Setup Wizard

The single image defaults to `node docker/run-all.js`. If it starts without `PO18_PG_URL` and without `/config/app.env`, it serves the setup wizard on port `3100`. You do not need to pass database or bot values in `docker run`; fill them in the wizard first.

The wizard can test the PostgreSQL connection, generate random secrets, show or hide secret fields, export `/config/app.env`, and exits after saving so Docker can restart into normal app mode.

Security tokens generated by the wizard:

- `PO18_UPLOAD_API_TOKEN`: required by external upload/write APIs through `X-Upload-Token` or `X-PO18-Upload-Token`.
- `PO18_BOT_API_TOKEN`: required by `/bot-api/*` through `X-Bot-Token`; empty values are rejected instead of being treated as open access.
- `PO18_METRICS_TOKEN` is optional. When set, `/metrics` requires `Authorization: Bearer <token>`.

Backups and restore:

- Database backups are stored in `/config/backups`.
- The admin System page can upload a PostgreSQL custom dump and restore it after typing `RESTORE <file>`.
- Before restore, the app automatically creates one more current database backup. After restore, the container exits so Docker can restart it with a clean connection pool.
- Long-running backup, restore, rank refresh, Bot export/sync/share, stale cleanup, and chapter-order repair tasks are visible in the admin Task Center.

Schema rollback is manual and disabled by default. After taking a backup, rollback the latest migration with:

```bash
docker exec po18-app sh -lc 'cd /app && PO18_ALLOW_SCHEMA_ROLLBACK=1 npm run db:rollback -- --steps 1 --confirm ROLLBACK'
```

Rollback everything newer than a known version:

```bash
docker exec po18-app sh -lc 'cd /app && PO18_ALLOW_SCHEMA_ROLLBACK=1 npm run db:rollback -- --to 005_system_jobs --confirm ROLLBACK'
```

The setup/config panel is protected by `PO18_SETUP_TOKEN`. If you do not provide one, the container prints a generated token in logs:

```bash
docker logs po18-app
```

Mount `/config` so the generated config survives restarts:

```bash
docker run -d --name po18-app --restart unless-stopped -p 3100:3100 -p 3200:3200 -v /opt/po18/config:/config wenmoux/reader:v1.0
```

Open `http://SERVER_IP:3100/setup?token=TOKEN`, save the PostgreSQL and bot settings, then wait for the container to restart. After config is present, the same container starts the backend on `3100`, reader on `3200`, and bot when `TELEGRAM_BOT_TOKEN` is configured.

After the app is running, the same protected panel is available at:

- `http://SERVER_IP:3100/setup?token=TOKEN` for config changes.
- `http://SERVER_IP:3100/setup/status?token=TOKEN` for server/reader/bot/database checks.
- `http://SERVER_IP:3100/setup/status?token=TOKEN` also includes redacted diagnostics with one-click copy.
- `http://SERVER_IP:3100/setup/logs?token=TOKEN` for recent container runtime logs, with filters such as `filter=error`, `filter=database`, `filter=bot`, and `filter=reader`.

The admin backend at `http://SERVER_IP:3100` is rebuilt with the same UI structure and visual style as the setup panel. The setup panel also exposes it at `http://SERVER_IP:3100/setup/admin?token=TOKEN`, so you can move from setup into the backend without using a separate entry point. After admin login, open the `System` tab to view service checks, filtered runtime logs, redacted diagnostics, a setup-panel shortcut, and a restart action.

If you are using an older command that explicitly appends `node docker/run-all.js`, it still works:

```bash
docker run -d --name po18-app --restart unless-stopped -p 3100:3100 -p 3200:3200 -v /opt/po18/config:/config wenmoux/reader:v1.0 node docker/run-all.js
```

For a local PostgreSQL container, start it before filling the wizard and use this DB URL:

```text
postgres://po18:po18-change-me@po18-postgres:5432/po18
```

## Status

Container health:

```powershell
docker compose ps
```

The single Docker image also includes a `HEALTHCHECK`, so `docker ps` shows `healthy/unhealthy` after startup.

Full status check from inside the compose network:

```powershell
docker compose --profile tools run --rm status
```

Local host status check:

```powershell
npm run status
```

Single-container status check:

```bash
docker exec po18-app node docker/status-check.js local
```

Recent runtime logs from the mapped config directory:

```bash
docker exec po18-app tail -n 200 /config/runtime.log
```

Health endpoints:

- `GET /health/ready` on `server-pg`: includes database connectivity.
- `GET /health/version` on `server-pg` or the setup wizard: returns app version, image tag, Node version, and platform.
- `GET /health/ready` on `reader`: checks the reader build is present.
- `GET /health/ready` on `bot`: checks Telegram startup and polling freshness.

## Sharing hygiene

The Docker build and git ignore rules exclude local runtime data, including `.env`, `.alma`, `.alma-snapshots`, `node_modules`, logs, local database files, generated booklist HTML, and old bundled reader builds. Do not publish a hand-made zip of the whole working directory unless you remove those ignored paths first.
